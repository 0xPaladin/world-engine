# Planet Types Implementation Plan

## Overview

Add a `planetType` selector (`earthlike` | `airless` | `barren` | `hostile`) to the Planet Generator that branches the simulation pipeline and colormap rendering. Default on load is `earthlike` (current behavior unchanged). The dropdown sits in the existing lil-gui "Planet" folder.

---

## Architecture

### Data flow

```
main.js GUI dropdown
  → planet.setPlanetType(type)
  → planet.generateMesh() / generateMap()
      → branches on _planetType inside generateMap()
        • earthlike:  plates(50% ocean) → standard elevation → FBM moisture → rivers
        • airless:    skip plates → crater elevation → zero moisture → no rivers
        • barren:     plates(all continental) → elevation + volcano boost → trace moisture → no rivers
        • hostile:    plates(high volcanism) → elevation + volcanic dominance → zero moisture → no rivers
  → rebuildAll() / applyClimate()
  → colormap-texture.js picks the right colormap array based on planet type
  → renderer skips rivers for non-earthlike, adds cloud sphere for hostile
  → applyPopulation() adapts or skips based on type
```

### Key invariants

- `mesh` (Fibonacci sphere + DualMesh) is identical for all types — only the **map data** differs
- `r_elevation` range stays roughly [-1, +1] for all types so colormap indexing works
- No new external dependencies

---

## Files to Modify

### 1. `colormap.js` — Palette generators per type

- Add a `planetType` parameter to `exports.data` — or export a function `getColormapData(type)` that returns the pixel array
- Current Earth colormap stays as-is (64×64, elevation×moisture → blue ocean / green land / white snow)
- **airless**: grayscale ramp across elevation; moisture axis has negligible effect. 4–6 gray bands for visual terrain layering. No blue band at negative elevation.
- **barren**: red/orange/ochre ramp; white band at high elevation (polar caps). Moisture axis muted. No blue band — trace deep red for lowest depressions.
- **hostile**: yellow/orange/brown ramp; no blue band; slight emissive-style brightness boost at highest temperature (elevation). Moisture axis flat.

### 2. `colormap-texture.js` — Dynamic texture

- Instead of building one texture at module init, expose `rebuildColormapTexture(type)` that:
  - Calls `colormap.getData(type)` to get the pixel array
  - Creates a new `DataTexture`, disposes the old one
  - Returns the texture for the shader uniform
- Called from `main.js` when planet type changes

### 3. `planet.js` — Simulation branching

- Add `_planetType = 'earthlike'` parameter with getter/setter
- Modify `generateMap()` with a `switch (_planetType)`:

  **`earthlike`**: unchanged current code path

  **`airless`**:
  - Skip `generatePlates()` entirely — all regions get `r_plate[r] = r` (single uniform plate) or just any sentinel
  - Use **crater-based elevation**: start with FBM noise baseline, then overlay explicit craters (choose random center points, subtract from elevation in disk, add rim). Seed from `_seed`.
  - `r_moisture` = 0 everywhere
  - Skip `assignTriangleValues`, `assignDownflow`, `assignFlow`
  - `quadGeometry.setMap()` still called but with zero moisture and no flow

  **`barren`**:
  - `generatePlates()` with all plates marked continental (no ocean in `plate_is_ocean`)
  - `assignRegionElevation()` works but `ocean_r` set collects only plate centers — no ocean regions
  - Post-process: apply **volcano boost** — randomly select a few regions, multiply elevation by 1.5–2.5
  - `r_moisture` = near-zero FBM clamped [0, 0.15], with slight polar enhancement (latitude > 60° → 0.2–0.4 for ice)
  - `assignTriangleValues` for consistent t-values, but skip `assignDownflow`/`assignFlow`

  **`hostile`**:
  - `generatePlates()` with all plates continental, but **increase P** internally by 1.5× for more smaller plates
  - `assignRegionElevation()` — boost `mountain_r` influence (thicker crust, more volcanic constructs)
  - Post-process: select random plate centers, apply large-radius dome elevation (+0.3–0.5) for volcanic rises
  - `r_moisture` = 0 everywhere
  - Skip `assignTriangleValues`/`assignDownflow`/`assignFlow` (or assignTriangleValues for rendering, but skip rivers)

### 4. `shaders.js` — Visual differentiation

- **earthlike**: current shader unchanged
- **airless**: use same shader; the grayscale colormap + high-contrast lighting gives the look. Optionally tweak `u_c` (ambient) down and `u_slope` up for harsher shadows
- **barren**: same shader; red colormap + thin atmosphere tint — optionally expose a `u_atmosphere_haze` uniform for a reddish rim glow (add in fragment shader)
- **hostile**: same shader for surface; **add a separate cloud sphere mesh**:

  ```js
  // Cloud sphere: slightly larger radius (1.01), translucent, noise pattern
  const cloudGeom = new THREE.SphereGeometry(1.01, 64, 64);
  const cloudMat = new THREE.ShaderMaterial({
    uniforms: { u_time: { value: 0 }, u_cloud_texture: { value: generateCloudTexture() } },
    vertexShader: '...', // pass position, normal
    fragmentShader: '...', // simplex noise cloud pattern, yellow-white, transparent
    transparent: true,
    depthWrite: false,
    side: THREE.FrontSide,
  });
  ```

  Cloud texture can be a 2D canvas with perlin/simplex noise, or generated in the fragment shader. Slowly rotates.

### 5. `renderer.js` — Conditional rendering

- `rebuildPlanet()` — pass `planetType`; no changes needed (geometry is identical)
- `rebuildRivers()` — **skip entirely** if `planetType !== 'earthlike'`
- Add `rebuildCloudSphere(type)`:
  - For `hostile`: create the cloud sphere mesh, add to scene
  - For all others: remove from scene if present
  - Called from main.js after rebuild
- Overlay rendering (culture/state/burg) — skip `applyPopulation()` for `airless` and `hostile` (no habitable surface)

### 6. `main.js` — GUI and wiring

- Add `_planetType = 'earthlike'` to `PARAMS`
- Add dropdown in the "Planet" folder:

  ```js
  fPlanet.add(PARAMS, 'planetType', ['earthlike', 'airless', 'barren', 'hostile'])
    .name('Planet Type')
    .onChange(v => {
      planet.setPlanetType(v);
      planet.generateMesh();  // calls generateMap internally with new type
      rebuildAll();
      applyClimate();
      renderer.rebuildCloudSphere(v);
      rebuildColormapTexture(v);  // update shader uniform
      setTimeout(() => window.applyPopulation(), 200);
    });
  ```

- On initial load: ensure `planetType = 'earthlike'` so colormap and shaders initialize correctly
- Add a **"New Planet" button next to the dropdown** (or modify the existing one to respect planet type)

### 7. `world-population.js` — Suitability adaptation

- `computeSuitability()` — currently uses moisture + coastal bonus + elevation
- For `airless` and `hostile`: make all cells unsuitable (score = 0) so no cultures/states spawn
- For `barren`: reduce moisture-based multiplier; keep coastal bonus; population is sparse
- Pass `planetType` from `main.js` → `generatePopulation()`

## Implementation Order

1. **colormap.js** — Add `getColormapData(type)` function with all four palettes
2. **colormap-texture.js** — Add `rebuildColormapTexture(type)` 
3. **planet.js** — Add `_planetType`, branch `generateMap()`, implement each type's elevation/moisture logic
4. **shaders.js** — Add cloud sphere shader for hostile type
5. **renderer.js** — Add cloud sphere management, conditional river skip, pass type to colormap
6. **world-population.js** — Add type-aware suitability
7. **main.js** — Wire GUI dropdown, initial state, regeneration flow

## Open Questions / Considerations

1. **Seed stability** — The `airless` type doesn't use plates, but still seeds from `_randomNoise` (FBM). Craters need their own seed derivation. Should work as long as `mesh` generation (Fibonacci sphere) stays deterministic from the same seed.

2. **QuadGeometry.setMap() for non-earth types** — The `coast` branch in the quad triangulation uses `r_elevation < 0`. For types with no oceans, what happens when all elevation is ≥ 0? The `else` (ridge) branch fires for all sides. That's fine for rendering — just no water.

3. **`pickRegion` biome labels** — Currently hardcodes Earth biome names. Should return type-appropriate labels: "Crater Floor" / "Highlands" / "Mare" for airless; "Regolith" / "Planitia" for barren; "Sulfurous Plain" / "Volcanic Rise" for hostile.

4. **Rendering performance** — Cloud sphere for hostile type is an extra draw call. Negligible impact with current vertex counts.

5. **Atmosphere rim lighting** — For barren (Mars-like), a subtle red atmosphere glow on the horizon could be done in the fragment shader via a rim-light term mixed with red. This is optional polish.

6. **Water Level slider interaction** — For `airless`/`barren`/`hostile`, the water level slider should be hidden or disabled (no oceans to submerge). Could show a tooltip or leave it as a no-op.

7. **Temperature/Rainfall sliders for non-earth types** — These modify the colormap lookup via elevation/moisture remapping. For `airless` (zero moisture), the rainfall slider is a no-op. Temperature still affects elevation scaling. We should decide: (a) keep sliders for all types (some dead zones), (b) show/hide dynamically per type.

**Recommendation for Open Questions 6–7**: Leave all sliders visible for simplicity. Their behavior degrades gracefully: water level changes nothing when there's no ocean (all elevation ≥ 0), rainfall changes nothing when moisture is 0, temperature still works for all types.
